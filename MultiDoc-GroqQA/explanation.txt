================================================================================
                    MULTI-DOCUMENT PDF QA APPLICATION
                         Technical Explanation
================================================================================

OVERVIEW
--------
This application allows users to upload multiple PDF documents and ask questions
about their content. It uses AI to find relevant information and generate
accurate answers based on the uploaded documents.

================================================================================

LIBRARIES USED AND WHY
----------------------

1. STREAMLIT
   - Purpose: Web application framework
   - Why: Creates an interactive web UI without writing HTML/CSS/JavaScript
   - Features used: File uploader, text input, sidebar, spinner, buttons

2. PyPDF2
   - Purpose: PDF text extraction
   - Why: Reads and extracts text content from uploaded PDF files
   - How: Iterates through each page and extracts text using extract_text()

3. LANGCHAIN-TEXT-SPLITTERS (RecursiveCharacterTextSplitter)
   - Purpose: Splits large text into smaller chunks
   - Why:
     * LLMs have token limits (can't process very long text at once)
     * Smaller chunks improve search accuracy
     * chunk_size=10000: Maximum characters per chunk
     * chunk_overlap=1000: Overlap between chunks to maintain context continuity

4. FAISS (Facebook AI Similarity Search)
   - Purpose: Vector database for similarity search
   - Why:
     * Extremely fast similarity search (milliseconds even with millions of vectors)
     * Memory efficient
     * Works locally without external database setup
     * Created by Facebook/Meta for production-scale applications
   - How it works:
     * Stores text chunks as numerical vectors (embeddings)
     * When user asks a question, finds most similar chunks using cosine similarity
     * Returns relevant chunks to provide context for the answer

5. HUGGINGFACE EMBEDDINGS (all-MiniLM-L6-v2)
   - Purpose: Converts text into numerical vectors (embeddings)
   - Why:
     * Free and runs locally (no API costs)
     * all-MiniLM-L6-v2 is lightweight (~90MB) but effective
     * Good balance between speed and accuracy
   - How: Transforms text into 384-dimensional vectors that capture semantic meaning

6. GROQ
   - Purpose: LLM API for generating answers
   - Why:
     * Extremely fast inference (uses custom LPU hardware)
     * Generous free tier
     * Access to powerful models like Llama 3.3 70B
   - Model used: llama-3.3-70b-versatile
     * 70 billion parameters
     * Excellent for question-answering tasks

7. PYTHON-DOTENV
   - Purpose: Environment variable management
   - Why: Securely stores API keys in .env file instead of hardcoding in source code

KEY CONCEPTS EXPLAINED
----------------------

WHAT ARE EMBEDDINGS?
- Embeddings are numerical representations of text
- Similar texts have similar embeddings (close in vector space)
- Example: "dog" and "puppy" would have similar vectors
- Used to find semantically similar content

WHAT IS VECTOR SIMILARITY SEARCH?
- Traditional search: Exact keyword matching
- Vector search: Finds semantically similar content
- Example: Query "automobile" can find text about "car" or "vehicle"
- FAISS performs this search very efficiently

WHY CHUNK THE TEXT?
- LLMs have context length limits
- Smaller chunks = more precise retrieval
- Overlap ensures no information is lost at chunk boundaries

WHAT IS RAG (Retrieval Augmented Generation)?
- This app uses the RAG pattern
- Retrieve: Find relevant chunks from FAISS
- Augment: Add retrieved chunks to the prompt as context
- Generate: LLM generates answer based on context
- Benefit: LLM answers based on YOUR documents, not just its training data

================================================================================

PARAMETERS EXPLAINED
--------------------

chunk_size=10000
- Maximum characters per chunk
- Larger = more context but less precise retrieval
- Smaller = more precise but may lose context

chunk_overlap=1000
- Characters shared between adjacent chunks
- Prevents information loss at chunk boundaries

temperature=0.3
- Controls randomness in LLM responses
- 0.0 = deterministic, same answer every time
- 1.0 = very creative/random
- 0.3 = balanced, mostly consistent but slightly varied

allow_dangerous_deserialization=True
- Required for loading FAISS index from disk
- Safe here because we created the index ourselves
